# $rotatebone / $movebone

Edit a bone's **rest/bind pose** late in the QC - after all geometry
(`$body`/`$bodygroup`/`$model`), animations (`$sequence`/`$animation`),
`$deltaproportions`, and `$definebone`/`$unlockdefinebones` have been loaded.

They relocate a joint's pivot/orientation **without deforming the mesh at rest**
and **without changing how existing animations look** - only the reference
(bind) pose moves. This is the re-rigging primitive: shift a pivot in place and
let the engine deform around the new pivot during animation.

There is a **combined budget of 16 uses** across both commands. The 17th
`$rotatebone`/`$movebone` is a hard error.

Place these commands at the end of the QC.

## Syntax

```
$rotatebone <bonename> <rx ry rz> [local | world] [ignorehitbox]
$movebone   <bonename> <x y z>    [local | world] [moveweight <residualbone> [factor]] [ignorehitbox]
```

## $rotatebone

Rotates the bone's bind orientation. The rotation pivots about the bone's own
origin (the bone position is never displaced). Angles use the same Euler
convention as `$definebone`.

- **`local`** *(default)* - `rx ry rz` are applied about the bone's local axes.
- **`world`** - applied about the bone's origin using world-aligned axes.

```
$rotatebone "ValveBiped.Bip01_L_UpperArm" 0 0 15
$rotatebone "ValveBiped.Bip01_Head1" 10 0 0 world
```

## $movebone

Translates the bone's bind position.

- **`world`** *(default)* - `x y z` are along world/model axes.
- **`local`** - along the bone's local axes.

```
$movebone "ValveBiped.Bip01_Pelvis" 0 0 2
$movebone "ValveBiped.Bip01_L_Hand" 1 0 0 local
```

### moveweight <residualbone> [factor]

Optional trailing clause on `$movebone` only. The bind-pose edit itself is
weight-preserving - it relocates the pivot without re-skinning anything - so on
its own a `$movebone` never changes which bones drive a vertex. `moveweight` is
the opt-in to actually **re-skin**: every vertex weighted to the moved bone has
`[factor]` of that influence **transferred to `<residualbone>`**, and its weights
are renormalized to 1.0. If the vertex is already weighted to `<residualbone>`,
the transferred share is merged into it.

`[factor]` is **optional** (default **`0.55`**) and clamped to `[0, 1]`:

- **`1.0`** - full transfer; the moved bone's influence moves entirely to the
  residual bone (the residual replaces it).
- **`0.5`** - half stays on the moved bone, half goes to the residual (a blend).
- **`0.0`** - no transfer (a no-op).

Omitting the float (legacy `moveweight <residualbone>`) uses the `0.55` default,
so existing QC keeps working.

Because the rest position was already baked from the original weights, the
**rest mesh stays visually identical** - only the animation deformation of the
re-skinned fraction now follows `<residualbone>` instead of the moved bone.

```
$movebone "ValveBiped.Bip01_L_Forearm" 0 0 -1 local moveweight "ValveBiped.Bip01_L_UpperArm" 1.0
```

> Note: `moveweight` is the experimental part of this feature. The default
> (no `moveweight`) path already preserves the mesh fully; validate `moveweight`
> on a flex-free test model before relying on it.

## How it works

Each edit is expressed as a change of the bone's bind frame and folded into the
bone's `srcRealign` transform. Because `srcRealign` is applied to **both** vertex
binding (`RemapVerticesToGlobalBones`) and every animation frame
(`ConvertAnimation`/`RemapAnimations`), the rest mesh and all animation visuals
stay consistent automatically. Direct children are re-expressed relative to the
new parent frame, so the subtree below the edited bone stays world-fixed at rest.
Delta animations and `$deltaproportions` flow through the same path and need no
special handling.

## Keeping hitboxes in place: `ignorehitbox`

By default a bone edit also moves that bone's hitboxes (they are rigidly attached
to the bone). Append `ignorehitbox` to a `$movebone`/`$rotatebone` to keep the
bone's **manual** `$hbox` hitboxes at their original world location - the compiler
compensates the hitbox's compile values by the inverse of the edit ("moving the
opposite"):

- A `$movebone` shifts the hitbox `bmin`/`bmax` the opposite way.
- A `$rotatebone` writes the inverse rotation into the hitbox's orientation so the
  box keeps its world orientation.

```
$movebone "ValveBiped.Bip01_L_Foot" 0 0 -2 world moveweight "ValveBiped.Bip01_L_Calf" 1.0 ignorehitbox
$rotatebone "ValveBiped.Bip01_Head1" 0 0 15 ignorehitbox
```

> `ignorehitbox` only affects **manually authored** `$hbox` hitboxes.
> Auto-generated hitboxes are recomputed from the edited skeleton and already stay
> wrapped around the (unmoved) mesh, so the keyword is a no-op for them.

## Opting specific animations out: `ignorebonemove` / `ignorebonerotate`

By default these bind-pose edits flow into **every** animation (the world motion
is preserved, only re-expressed relative to the new bind frame). This includes the
`a_reference`/`a_proportions` animations generated by
[`$deltaproportions`](deltaproportions.md).

For the proportion-delta (`subtract`) trick you often do **not** want a rig
adjustment like a foot `$movebone` baked into the proportion delta - the offset
already lives in the bind pose, and baking it into the delta would double-apply
it. Two opt-in per-animation flags suppress the edits for a given animation:

- **`ignorebonemove`** - convert this animation as if `$movebone` edits never happened.
- **`ignorebonerotate`** - convert this animation as if `$rotatebone` edits never happened.

They are valid anywhere `$animation` options are accepted: directly on
`$animation`, inside a `$sequence` block, and inside `$deltaproportions`'s
`appendreference { ... }` / `appendproportions { ... }` blocks.

For a clean `subtract`, set the flag on **both** the reference and the proportion
animation:

```
$movebone "ValveBiped.Bip01_L_Foot" 0 0 -2 world moveweight "ValveBiped.Bip01_L_Calf" 1.0
$movebone "ValveBiped.Bip01_R_Foot" 0 0 -2 world moveweight "ValveBiped.Bip01_R_Calf" 1.0

$deltaproportions {
    referencepose  "female_01_reference.smd" 0
    proportionpose "faelynn_default.dmx"     0

    appendreference   { ignorescale noautoik ignorebonemove }
    appendproportions { ignorebonemove }
}
```

The bind pose and the regular (unflagged) gameplay animations still show the
lowered feet; only the proportion delta excludes the move.

## Behavior notes

- Bones are resolved after `$renamebone` and bone collapse, so a name that was
  collapsed away (or never existed) warns and is skipped - the compile continues.
- Multiple edits compose in QC order; editing a parent then a child is supported.
- A root bone (no parent) is valid.
- Affects the `-definebones` export, since edits are applied before that dump.
- Ragdoll collision (`$collisionjoints`) follows the edits: a moved/rotated bone's
  physics hull stays aligned with its mesh. Single-body `$collisionmodel` is a rigid
  model-space hull and is unaffected by per-bone edits.
